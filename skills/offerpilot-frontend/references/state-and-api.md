# 状态与接口

## 表单

当前阶段默认使用：

- `react-hook-form`
- `zod`

分工：

- `react-hook-form` 负责表单状态、提交和错误管理
- `zod` 负责输入校验和结构约束

适用场景：

- 登录
- 注册
- JD 输入
- 用户资料编辑

## 数据请求

查询型数据默认优先用轻量方案。

建议：

- 普通请求先封装到 `services/`
- 查询类状态优先用 `SWR`
- 页面不要直接到处写 `fetch`

例如：

- `services/auth.ts`
- `services/users.ts`
- `services/jobs.ts`

## 登录态

当前阶段先保持简单。

建议：

- 后端返回 token 后，前端统一保存
- 请求时统一带上 `Authorization: Bearer <token>`
- 当前用户信息通过 `/users` 这类接口读取

如果没有更强的安全要求，第一版可以先用较简单的浏览器存储方案。

后面如果要提升安全性，再评估更稳妥的服务端会话或 cookie 方案。

## 状态边界

- 表单输入状态交给表单库
- 查询型远程数据交给 `SWR`
- 页面局部交互状态用 React 自己的 state
- 不要一开始就引入重型全局状态管理

## 当前阶段不建议

- 一开始上 Redux 这类重型方案
- 页面内直接混杂接口、校验和复杂视图逻辑
- 登录态方案频繁切换
- 为未来而未来地设计很重的 API 封装层
